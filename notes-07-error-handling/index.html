Error Handling

The first rule to error handling as a programmer is to understand it is never the browser or the computer's fault. It is always your fault.

Step 1: Check your File Links

The first step to resolving errors is to check your files and links to ensure that they are properly connected to your files and that they are in the right place

In Brackets or Sublime Text, you need to ensure that you open the file folder instead of the individual files. Also in Sublime/Brackets, if you save a file or move a file after saving it, the program will create an old copy of your file prior to moving it.

If you move your files around in your file folder. or move your files outside of your original file folder always make sure to close Sublime/Brackets then re-open the file folder in your code editor. Do NOT save before you close the code editor after moving your files.

Step 2: Check the Order That You are Linking Files

The order with which you link files matters. In HTML, CSS and JS, the file linked last will overwrite all files that come before it. This means that you should ensure that your custom CSS file and your custom JS file are linked after framework/library files such as bootstrap.css, normalize.css or jQuery Library.

Step 3: Weed Out Common Errors or Mistakes

The mast majority of beginner and experienced programmer mistakes are typos. Always check the following for typos or problems before attempting a more complex fix...

  • Check your file names, tag names, attribute names, values, etc. Are they spelled correctly? Is your code editor highlighting a certain chunk of code wrong?
  • Have you opened and closed your tags or code blocks properly?
  • Are you looking at the right file?

Step 4: Using the Console and Inspector in Your Browser

In Firefox and Chrome, you access the Console and Inspector by right-clicking on the screen and selecting "Inspect" or "Inspector".

The Element Inspector is for HTML and CSS debugging. The Console is for javascript debugging.

The Element Inspector

The Element Inspector, used to debug HTML and CSS can be useful for finding or working with the following scenarios...

  • Testing properties and values to see immediate effects.
  • Finding specific classes or IDs used in order to change or modify an element.
  • Testing a selection/selector without modifying the original code files.
  • Toggling on or off properties and rules to determine changes or effects.

The Console

Javascript utilizes many of the browser's developer tools, the console is the simplest to use for debugging purposes.

Usually, the console will attempt to find and flag errors that have to do with syntax or improper typing. It will not find everything wrong about a code file. If you find that your code is breaking at certain points, try using a console.log() method to see where your code happens to be breaking.

Step 5: Use Validators and/or JSHint

There are tools available online that will attempt to find and call out errors for you. They are not perfect and will not catch all errors, however, they can be helpful when you just can't find where the typo is occuring.

  • HTML Validator: W3C HTML Validator
  • CSS Validator: W3C CSS Validator
  • JS Hint: JSHint

Step 6: Commenting Out Code

If Console and JSHint are unhelpful, you may wish to start commenting out the chunks of code that are giving you trouble. As you comment out the code, you may wish to comment large swathes to start, then slowly uncomment smaller bits of code until you find the area where the problem is occuring.

Step 7: Ask Somebody

The last ditch resort to fixing or debugging code is to ask someone else. Before you ask another human being, attempt to find someone who has already experienced your problem by using Google.

If Google is unhelpful or if you cannot find the right search terms to use, try to ask someone else with some coding knowledge to look at your program. This can be someone you know or a coding help forum such as StackOverflow or r/learnjavascript
